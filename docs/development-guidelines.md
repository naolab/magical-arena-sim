# 開発ルール & Git 運用ガイド（v0.1）

## 基本方針
- すべての作業は新しいブランチを作成してから着手する。`main` 直コミットは禁止。
- Pull Request（PR）でレビュー・確認を行い、マージ時には必ずマージコミットを作成する。
- 変更内容はドキュメント化し、仕様・要件との整合性を常に維持する。

## ブランチ命名規則
```
<種別>/<短い説明>
```
- 種別の例:
  - `feat` : 新機能
  - `fix` : バグ修正
  - `refactor` : 構造改善
  - `docs` : ドキュメント変更
  - `chore` : ビルド・開発環境調整 など
- 例: `feat/battle-simulation-ui`, `fix/damage-calculation`, `docs/update-spec-v12`

## コミットメッセージ規約
```
タグ:概要
```
- タグは以下を基本とし、英語（小文字）で記述する。
  - `feat` / `fix` / `refactor` / `docs` / `style` / `test` / `chore` / `ci`
- `概要`部分は日本語で書く。
- 例: `feat:バトル開始UIを実装`, `fix:アンチゲージ更新処理を修正`

## Pull Request 運用
- PR タイトルはブランチ名と同等の意味がわかる表現にする（例: `feat: バトルロジックの初期実装`）。
- 内容説明欄に:
  1. 目的・背景
  2. 主要な変更点
  3. 動作確認結果
  4. レビュワーへの補足（任意）
- レビュー前に lint / test / build など必要なチェックを実行して結果を記載。
- マージは「Merge Commit」を使用し、Squash / Rebase は基本禁止。

## 作業フロー
1. `main` を最新化し、作業ブランチを作成。
2. 実装・テスト・ドキュメント更新をブランチ上で実施。
3. コミットは小さく分割し、タグ付き日本語メッセージで残す。
4. PR を作成し、レビュワーに共有。
5. レビュー対応後、CI が通っていることを確認してマージコミットで `main` に統合。

## その他ルール
- lint / format の自動化設定がある場合はコミット前に実行。
- レビューで指摘された内容は原則対応済みとして再レビューを依頼。
- 緊急対応で直接 `main` を変更する必要がある場合は、事後に必ず関係者へ共有し、対応内容をドキュメント化。
- ドキュメントや設定変更のみの場合でもブランチ運用・PR 作成の手順は同様に守る。
